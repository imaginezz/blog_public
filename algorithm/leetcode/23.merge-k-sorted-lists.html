LeetCode-Merge K Sorted Lists

  • ALGORITHM
  • LEETCODE
  • 23.MERGE-K-SORTED-LISTS

https://leetcode-cn.com/problems/merge-k-sorted-lists/

又是拖了一天才开始做 Orz。
首先，用最笨的办法，每次从三个数字里面挑出来一个最小的，添加到一个新的序列里去。复杂度为 O(kN)，所以如我所料的超时了。

接下来的思路和之前 Merge Two Sorted Lists很像 ，将序列两两进行合并。

但是在文末的很变态的 test case 下，还是超时了。

现在只能看看 solution 了。

Solution 中的第一个方法居然是直接将所有数字排序，排序结束后重新生成各个 Node。这样。。居然还挺快（排序复杂度 O(NlogN)），空间复杂度也就是 O(N)，完全可接受，真的是好办法。

第二种方法和我的第一种方法一样，由于复杂度太高，没法 Accept，所以根本没提供代码。不过它提供了使用了 PriorityQueue 的代码。

这段代码的设计也比较巧妙，利用优先队列可以排序的特性，现将队列中分别插入这几个 List 的头部，然后每出一个最小的，就将当前这个 List 前进一位，并插入队列中进行排序，然后再次输出。

这个算法使用了小根堆，的时间复杂度可以到 O(Nlogk)，空间复杂度可以到 O(K)。

Solution 中的第三种方法是我上面的第二种方法，复杂度和第一种一样高（只是简化了几个步骤，复杂度没变）。

最后一个方法是上一种方法的变种，它使用了分治的思路可以将几个 List 两两合并，可以参考图

这样，每一轮合并的序列都比之前有序，时间复杂度就会降低为 O((2N)*(K / 2)+(4N) *(K/4)+(8N)*(K/8)+...+(2^logK*N)*(K/(2 ^logK)))=O(logK*KN)。

所以，在思考的时候还是要多积累，不一定非要追求那么好的空间复杂度，有时候可以为了时间，牺牲一些空间（Solution 中的第一种方案）。
